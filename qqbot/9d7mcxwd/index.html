开发框架

1640字约5分钟

2024-09-05

0. 前言

接下来的这篇文章会带你弄懂框架整合TypeScript开发的规则。相比于OneBot原生的API，整合之后的框架在开发一些常见业务的时候会非常方便灵活

在"快速开始"部分，我们已经初步了解了框架提供的一个初始文件"main.ts"的作用。这个文件非常重要，里面提供了可以控制机器人上线和下线时候的行为的方法。除此之外，这个文件还是整个程序的入口，下面所有用于业务注册的文件都需要导入"main.ts"这个文件，否则业务注册是不生效的

基于上面这个前提，接下来我们可以开始开发自己的业务了

1. 业务注册

对于常见的一些事件，比如说私聊和群聊，整合版的框架已经提供了mapper装饰器让你快速实现对应的功能，例如：

import { mapper, LagrangeContext, PrivateMessage } from 'lagrange.onebot';

export class ... {

    @mapper.onPrivateUser(1193466151)
    async handleTestPrivate(c: LagrangeContext<PrivateMessage>) {
        // ...
    }

    @mapper.onGroup(956419963)
    async handleTestGroup(c: LagrangeContext<GroupMessage>) {
        // ...
    }
}

代码说明：

  • 非常重要的一点：千万别忘记导包！

  • 以@mapper.onPrivateUser(1193466151)为例，这个装饰器帮你注册了一个和QQ号为1193466151的账号的私聊业务，也就是说，其对应的handleTestPrivate函数下的所有逻辑，只有在QQ机器人的账号和1193466151这两个账号之间的私聊会话才会生效

  • 同理，@mapper.onGroup(956419963)则注册了一个只对956419963这个QQ群有效的业务，对于其它任何私聊会话或者群号不为956419963的QQ群都不生效

  • "c"是一个类型为LagrangeContext的，用于代指上下文的变量。LagrangeContext这个对象里面封装了很多开箱即用的API，比如之前示例一直在用的sendPrivateMsg

对于注册群聊事务的装饰器，还可以添加一个额外的参数：

@mapper.onGroup(956419963, {at:true})

这里的{at: true}的意思是只有当群里的人@机器人的QQ号的时候，下面的业务代码才会生效。这样设计的好处是，可以避免群友无意触发程序中设定的一些事件，或者导致机器人大量地做一些无意义的回复

2. 插件的注册使用

为什么要用插件

对于一些常用的功能，这个整合版的框架提供的插件注册功能可以大大提升一些常用代码逻辑的复用性，让开发者没必要在每一个注册的事件中去实现它们。比方说我要实现一个功能，给QQ机器人接入大模型，在和任何人或者群的聊天场景下，我都需要保证这个大模型业务是可用的，这个时候就可以给大模型功能注册一个插件，而不需要每个事件中都去写一遍大模型对应的业务代码

内置插件

框架的内置插件只有两个，仅用于测试

其中一个插件叫做echo插件，可以让QQ机器人重复你说的话。如果你想给某个事件注册这个插件，写法如下：

import { mapper, LagrangeContext, PrivateMessage, plugins } from 'lagrange.onebot';

export class ... {

    @mapper.onPrivateUser(1193466151)
    @plugins.use('echo')
    async handleTestPrivate(c: LagrangeContext<PrivateMessage>) {

    }
}

注意，这里别忘记导包！

仅仅需要一行@plugins.use('echo')的注册插件代码就可以让你的被注册事件实现让机器人重复你说的话

这样一来，每当这个用户使用 \echo xxx 和机器人对话时，机器人就会重复 xxx 的内容：

自定义插件

想要自定义插件，则需要使用全局对象 plugins 的 register 装饰器。比如我们这里实现一个简单的图片链接下载器：

User: \wget-image https://picx.zhimg.com/80/v2-bdae55043d61d7bcfeeabead6e953959_1440w.jpeg
Bot: <图片>

注册插件的事务函数的函数签名必须是：

(c: LagrangeContext<Message>) => any;

我们先在项目原有文件的基础上新建一个 plugins.ts 文件。

// plugins.ts

import { plugins, LagrangeContext, Message } from 'lagrange.onebot';

export class Impl {
    // 用"@plugins.register"来注册插件
    @plugins.register('wget-image')
    async wgetImage(c: LagrangeContext<Message>) {
        // 判断一下，只解析 message 类型的数据
        if (c.message.post_type === 'message') {
            const text = c.message.raw_message;
            if (text.startsWith('\\wget-image')) {
                const url = text.substring('\\wget-image'.length).trim();
                c.sendMessage([
                    {
                        type: 'image',
                        data: {
                            file: url,
                            timeout: 10000 //设置下载图片的超时时间
                        }
                    }
                ]);

                // 插件中使用 finishSession 会让被装饰的事务函数不再被执行，直接结束对话
                c.finishSession();
            }
        }
    }
}

在 impl.ts 中安装它：

//不要忘记导入插件对应的文件
import './plugins';
import { mapper, plugins, LagrangeContext, PrivateMessage, GroupMessage, Send } from 'lagrange.onebot'

export class Impl {
    @mapper.onPrivateUser(1193466151)
    //使用插件，括号里面的参数是插件的名字，必须和注册的时候用的名字保持一致
    @plugins.use('wget-image')
    async handleTestPrivate(c: LagrangeContext<PrivateMessage>) {
        c.finishSession();
    }
}

然后就可以实现用户输入"\wget-image xxx"(xxx是图片的url路径)的时候，bot会自动帮你把url路径转换成图片：

3. 开发建议

这个整合TypeScript的框架提供的初始代码是很简单的，但是如果你准备开发一个功能强大的Bot，项目代码会越来越庞大，因此对于代码合理的管理是非常有必要的。对此我提出一些个人建议：

  • 将main.ts作为程序的主入口，每次编译运行main.ts即可，对于一些其它的事件注册和业务逻辑可以放在其它文件里面，然后统一导入main.ts文件中
  • 对于不同功能的文件进行分类管理，例如对于事件注册的文件可以专门放到一个文件夹里面；对于一些自己写的功能性函数可以放到一个专门的工具类文件夹里面……
  • 对于一些常用的功能多使用框架提供的插件功能，我个人认为这个框架的插件功能的设计是很好的，可以大大降低代码的耦合度，便于开发者对项目进行管理
